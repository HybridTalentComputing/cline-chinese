## 工作原理

默认情况下，Cline 在调用任何工具（包括读取或写入文件）之前都会请求您的许可。

如果您想允许 Cline 在不询问的情况下执行某些操作，您可以为该工具设置自动批准权限。

## 权限选项

-   **读取项目文件**

    -   允许 Cline 在不询问的情况下读取当前工作区内的文件
    -   **读取所有文件**
        -   将读取权限扩展到工作区外的文件（系统文件、配置文件等）

-   **编辑项目文件**

    -   允许 Cline 在不确认的情况下修改当前工作区内的文件
    -   **编辑所有文件**
        -   将修改权限扩展到工作区外的文件

-   **执行安全命令**

    -   允许执行模型认为非破坏性的终端命令
    -   **执行所有命令**
        -   允许执行任何终端命令而不询问

-   **使用浏览器**

    -   允许 Cline 使用浏览器工具获取网络内容

-   **使用 MCP 服务器**

    -   允许连接和使用 MCP 服务器以获得扩展功能

-   **最大请求数**
    -   设置 Cline 在需要您输入之前可以连续执行的自动化操作数量

## 最佳实践

个人而言，我喜欢保持自动编辑禁用，因为它让我有机会在每一步都审查更改。

对于大多数严肃的开发工作流程，我建议从以下开始：

-   自动批准对项目文件的读取访问
-   设置合理的最多请求限制（10-20）

这给 Cline 足够的自由来探索您的代码库，而不会不断中断，同时仍然需要编辑或潜在破坏性操作的许可。

随着您对 Cline 在特定项目中能力的信任增加，您可以逐渐增加权限以匹配您的舒适度。

请记住，您始终可以根据需求变化调整这些设置 - 为关键生产工作收紧权限，或在原型设计和探索时放宽权限。

您甚至可以使用快速"星标"操作来快速切换您的自动批准选择。

## 使用建议

-   **开发阶段**：可以适当放宽权限，让 Cline 更自由地工作
-   **生产环境**：建议收紧权限，确保代码安全
-   **团队协作**：根据团队成员的技能水平设置不同的权限
-   **项目类型**：原型项目可以更宽松，正式项目需要更严格的控制

## 安全考虑

-   自动批准功能虽然方便，但也带来一定的安全风险
-   建议定期审查自动批准设置
-   对于敏感项目，建议保持手动批准模式
-   使用检查点功能作为额外的安全网
