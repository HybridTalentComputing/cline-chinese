`/newtask` 是一个斜杠命令，就像一个完美的开发者交接。它智能地打包重要的内容 - 整体计划、已完成的工作、相关文件和下一步 - 到一个具有干净上下文窗口的新任务中。同时留下工具调用、文档搜索和实施细节的噪音。

这正是当您将新开发者引入项目时会做的事情：提供他们继续工作所需的基本上下文，而不会用之前的每个按键操作压倒他们。

#### 使用 `/newtask` 斜杠命令

当您的上下文窗口正在填满但您还没有完成项目时：

	<img
		src="https://storage.googleapis.com/cline_public_images/docs/assets/newtask.png"

-   在聊天输入字段中输入 `/newtask`
-   Cline 将分析您的对话并提出要携带的上下文的提炼版本
-   您可以在提交之前通过对话完善这个提议的上下文
-   一旦满意，会出现一个按钮来创建具有您完善上下文的新任务

#### 示例

我经常在通过多步骤的复杂实施时使用 `/newtask`。例如，如果我完成了 10 步过程中的 3 步，而我的上下文已经 75% 满了文档片段、文件内容和详细讨论。

而不是失去这些见解或从头开始，我使用 `/newtask` 让 Cline 提取重要的内容 - 关键决策、文件更改和迄今为止的进度 - 而没有所有单独工具调用和研究步骤的噪音。

我喜欢将 `/newtask` 想象为一个新开发者加入项目。我需要让他们完全理解已完成的工作，了解相关文件，任何其他有用的上下文，以及下一步要去哪里。

#### 灵感

以下是一些使用 `/newtask` 的流行方式：

-   我使用 Context7 MCP 服务器研究复杂的 API，用文档填充我的上下文。一旦我理解了概念，我就使用 `/newtask` 重新开始，只包含实施所需的基本知识。
-   在通过多次调试尝试和文件探索识别出困难错误的根本原因后，我使用 `/newtask` 继续，包含解决方案但丢弃所有失败尝试的干净状态。
-   当客户讨论探索多种方法并最终确定一个方向时，我使用 `/newtask` 专注于实施选定的解决方案。
-   对于跨越多天的复杂项目，我在逻辑停止点使用 `/newtask` 以保持干净的工作空间，同时携带我的进度。
