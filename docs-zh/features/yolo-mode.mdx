YOLO 模式正如其名 - Cline 自动批准所有操作。在功能设置中勾选复选框，他将执行文件更改、终端命令，甚至从计划模式到执行模式的转换，而无需询问。

将其视为[自动批准](/features/auto-approve)的增强版 - 与细粒度权限不同，YOLO 模式赋予 Cline 完全自主权。

<Note>
**警告：这很危险。** YOLO 模式禁用所有安全检查。Cline 将不经许可执行他决定的任何操作。
</Note>

## 什么会被自动批准

当启用 YOLO 模式时，Cline 自动批准：

- **所有文件操作** - 在系统任何位置读取、写入和修改文件
- **所有终端命令** - 包括潜在破坏性操作
- **浏览器操作** - 网页抓取、表单提交、导航
- **MCP 服务器工具** - 外部集成和 API 调用
- **模式转换** - 从计划模式到执行模式的自动切换

基本上，移除了所有安全护栏。Cline 以完全自主权运行。

## 如何启用 YOLO 模式

导航到 Cline 设置 → 功能并勾选"YOLO 模式"复选框。就这样 - 没有确认对话框，没有额外警告。一旦启用，Cline 将立即开始自动批准所有操作。

要禁用它，只需取消勾选复选框。一旦关闭 YOLO 模式，任何待处理的操作仍需要你的批准。

## 何时使用它

YOLO 模式主要为我们要到来的可编写脚本的 CLI 而构建，其中完全自主执行是合理的。在 GUI 中，你可以考虑将其用于：

**快速原型设计**，你想要零摩擦并且不关心潜在错误。对于一次性实验或快速探索新想法非常完美。

**可信的重复任务**，你已经验证了 Cline 的方法并希望消除批准开销。考虑常规重构或既定模式。

**演示目的**，你希望在不不断中断的情况下展示 Cline 的能力。

## 可能出现什么问题

由于 YOLO 模式移除了所有安全检查，Cline 可能会：

- 在无警告的情况下删除重要文件
- 执行修改系统设置的命令
- 向外部服务发出网络请求
- 覆盖配置文件
- 安装或卸载软件包
- 提交并将更改推送到版本控制

风险级别完全取决于你要求 Cline 做什么。简单任务相对安全，但复杂请求可能会产生不可预测的后果。

## 最佳实践

如果你决定使用 YOLO 模式：

**从隔离环境开始。** 首先在一次性项目或沙盒环境中使用它。在理解风险之前，不要在生产代码库上启用它。

**对请求要具体。** 模糊的指令结合无限权限可能导致意外结果。你的要求越清晰，Cline 的行为就越可预测。

**监控输出。** 虽然 Cline 不要求许可，但他仍然向你展示他在做什么。观看终端输出和正在进行的文件更改。

**保持版本控制。** 确保你可以轻松恢复更改（如果出现问题）。当 YOLO 模式是你的工作流时，Git 成为你的安全网。

## 灵感：什么是可能的

启用 YOLO 模式后，你可以：

**从单个提示构建整个应用程序。** 描述你想要的内容，让 Cline 处理一切 - 文件创建、依赖安装、配置设置，甚至部署脚本。

**自动化复杂工作流**，通常需要数十次批准点击。数据处理管道、构建系统设置或多步骤重构操作变得无缝。

**快速迭代周期**，你可以在没有批准摩擦的情况下快速测试想法。非常适合探索不同方法或试验新技术。

**实时演示**，你可以展示 Cline 的完整能力，而无需停下来批准每个操作。非常适合演示或教学场景。

关键是理解 YOLO 模式将 Cline 从交互式助手转变为自主代理。明智地使用这种力量。
